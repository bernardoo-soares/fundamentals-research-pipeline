# Resume - 2026-03-15

Date: 2026-03-15

## Objective Clarified

This project is not primarily a data-pipeline project.
The real objective is to build a concentrated stock-picking research framework
for names investable today, using long historical operating data mainly as
context for quality, resilience, capital allocation, and trend analysis.

Key clarification:
- `2006-2025` history is mainly for understanding current companies.
- The project should not be framed as "build a cheap Bloomberg replacement".
- The project should not be framed as a broad historical alpha backtest on the
  current S&P 500, because that would create survivorship-bias issues.

## Agreed Strategic Direction

### Core Idea

Treat the pipeline as infrastructure, not the final project.

The real project should become:
- a concentrated, cross-sector stock-picking framework,
- with explicit score components,
- manual research input,
- sector-aware logic,
- and a confidence-aware layer for noisy/imperfect fundamentals.

### Target Style

The intended end state is closer to:
- "deep research lab for 10-30 names"

and not:
- "generic broad quant ranking engine"
- "full-universe factor backtest"
- "ML stock picker trained on noisy fundamentals"

## Research Philosophy Agreed

### ML vs Heuristic

Initial direction:
- do **not** start with ML.

Reasoning:
- the project is concentrated,
- labels are noisy,
- data is imperfect,
- explainability matters,
- and ML is likely to overfit early.

Preferred path:
1. Start with structured heuristic scorecards.
2. Build component-level scores first.
3. Later calibrate weights statistically.
4. Use light statistical learning only after the framework exists.

Important nuance:
- fixed weights can be provisional scaffolding,
- but final weights should ideally be learned/calibrated from research rather
  than treated as arbitrary permanent constants.

### Scoring Logic

The framework should use:
- a common cross-sector scoring spine,
- sector-specific modules,
- and a confidence-aware overlay.

Suggested component families:
- business quality
- capital allocation
- balance sheet resilience
- earnings quality
- valuation support
- confidence / thesis clarity

Important modeling note:
- avoid brittle hard thresholds where possible,
- prefer smoother scoring logic, sector-relative context, trends, and
  interpretation rather than naive ratio cutoffs.

Example:
- high ROE is not automatically good,
- because it may be leverage-driven or buyback-distorted.

## Universe Decision

### Current Conclusion

Using the **current S&P 500** is acceptable **if** the goal is:
- to study companies investable today,
- and to use `2006-2025` history only as operating context for those names.

Using the current S&P 500 is **not** appropriate for:
- historical predictive claims,
- full historical backtests,
- or learning broad predictive weights intended to represent real historical
  opportunity sets.

### Practical Universe Policy

Recommended near-term setup:
1. Use current S&P 500 as the default candidate pool.
2. Optionally add a small manual extension list of obvious large-cap names if
   they matter to the research process.
3. Narrow that down into a curated research universe.
4. Deeply score only 10-30 names.

### Historical Data Use

Use `2006-2025` history to derive indicators such as:
- long-run revenue growth
- margin stability
- free-cash-flow consistency
- dilution history
- debt behavior across cycles
- capital allocation behavior
- resilience in stressed periods
- recovery after drawdowns in operating performance

## Data Architecture Direction

### Recommended Storage

Do not use CSV as the main research store anymore.
Keep CSV mainly for exports/debug artifacts.

Recommended stack:
- Parquet + DuckDB as the core research storage/query layer.

Reasoning:
- better for evolving analytical work,
- better typing and compression,
- easier joins and time-series analysis,
- lower overhead than Postgres at this stage.

Current recommendation:
- use DuckDB now,
- postpone Postgres unless a real shared app / annotation UI is needed later.

### Suggested Table Families

The next architecture should evolve toward tables such as:
- `companies`
- `fact_observations`
- `fact_reconciled`
- `feature_quarterly`
- `score_component`
- `score_total`
- `manual_annotations`
- `research_notes`
- `thesis_log`

Important grain:
- company
- fiscal quarter
- as-of date
- source
- canonical metric
- value
- confidence

The `as-of date` / point-in-time dimension was identified as important.
